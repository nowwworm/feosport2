# План переделки FeoSport2 → Система мониторинга гонок дронов

**Источник требований:** Приказ Минспорта РФ № 1063 от 28.10.2024 «Правила вида спорта "гонки дронов (БВС)"»
**Базис:** существующая кодовая база FeoSport2 (auth, RBAC, competitions, heats, pilots, results, WebSocket, 100+ тестов, Windows-инсталлятор)
**Цель:** официально-совместимая платформа для проведения гонок по 8 дисциплинам ВРВС

---

## 1. Карта соответствия: что уже есть → что нужно

| Реализовано | Назначение в новой системе | Действие |
|---|---|---|
| Auth + JWT | Базовая аутентификация | Оставить как есть |
| RBAC: admin / chief_judge / judge / pilot | Соответствует ГСК (главный судья, заместитель, судьи) | Расширить: добавить роли `secretary`, `tech_judge`, `pit_judge`, `chronometer`, `informer`, `tech_director`, `doctor`, `team_representative`, `coach` |
| `competitions` | Соревнование | Расширить: статус, дисциплина, система проведения (2/4 или 4/8), формат (offline/online), стадия |
| `pilots` | Спортсмены | Расширить: пол, возрастная группа, медицинский допуск, ГТО, страховка, разряд/звание, регион/команда |
| `heats` + `heat_participants` | Вылеты | Переименовать концептуально в `flights` (вылеты), привязать к этапу и группе |
| `results` | Результаты | Расширить: круги, время по кругу, фальстарт, нарушения, штрафы, перелёты |
| WebSocket (score_update, leaderboard) | Реал-тайм судейство | Расширить событиями: `lap_complete`, `falsestart`, `dnf`, `pit_stop`, `channel_swap`, `disconnect` |
| Webhook регистрации пилотов | Интеграция с FormDesigner | Расширить: предварительная заявка по форме Приложения 2 |
| Тесты (100+) | CI/CD | Адаптировать после миграции схемы |

**Принцип:** не делаем «всё с нуля». Существующие сущности — фундамент. Добавляем новые таблицы и поля, мигрируем данные через ALTER, а не DROP.

## 2. Доменная модель (новые/расширенные сущности)

### Новые таблицы

```
disciplines              — справочник 8 дисциплин ВРВС
race_systems             — "2 of 4", "4 of 8"
race_formats             — "offline", "online"
stages                   — квалификация / 1/8 / 1/4 / 1/2 / финал
groups                   — группы внутри этапа (по 4 или 8 пилотов)
teams                    — команды (для командных дисциплин)
team_members             — связь pilot ↔ team (роль: pilot/mechanic/reserve, является ли капитаном)
drones                   — экземпляры дронов (вес, диагональ, видеоканал, серийник)
drone_specs              — технические требования по классу (валидация дрона)
video_channels           — каналы 5.8 ГГц для управления конфликтами
equipment_inspections    — записи комиссии по допуску (drone_id, judge_id, passed, notes)
admissions               — записи комиссии по допуску участников (документы, медицинский допуск)
laps                     — круги внутри вылета (flight_id, lap_number, duration_ms, valid)
falsestarts              — фальстарты
reflights                — перелёты (причина, инициатор)
pit_stops                — заходы в пит-стоп (для команд)
penalties                — штрафы (тип, основание, пункт правил)
protests                 — протесты (5-мин окно, текст, решение)
disconnects              — для симулятора (длительность, попытки)
documents                — загруженные документы (паспорт, страховка, согласие родителей)
age_groups               — справочник 10-17, 17-25, 14+
report_protocols         — итоговые протоколы (9 типов из Приложений 4, 5)
```

### Расширения существующих

```
users:
  + judge_role (для детализации судейских должностей)
  + qualification_category (ВК/1К/2К/3К/б/к)
  + region

pilots:
  + gender, birth_date, age_group_id
  + sport_rank (КМС, 1р, 2р, 3р, б/р)
  + gto_passed (boolean)
  + medical_clearance_until (date)
  + insurance_until (date)
  + region, team_id, registration_number

competitions:
  + discipline_id, race_system_id, race_format_id
  + status: draft → registration → admission → qualification → group → final → completed
  + age_group_id, gender (для разделения по полу/возрасту)
  + playoff_size: 16 / 32
  + venue_address, organizer_id
  + entry_fee, registration_deadline

heats (переименовать в flights в API, оставить таблицу heats для миграции):
  + stage_id, group_id, flight_number_in_stage
  + scheduled_at, started_at, ended_at
  + status: scheduled → in_progress → completed → reflight_required → cancelled
  + lap_limit | time_limit_seconds (одна из двух систем подсчёта)

results:
  + total_laps, best_lap_ms, total_time_ms
  + dnf, dsq (boolean), dnf_reason
  + points (баллы за вылет)
  + place
```

## 3. Поэтапный план реализации

### Фаза 0 — Подготовка (1-2 недели)

**Без кода. Дизайн и согласование.**

- [ ] Утвердить схему миграции БД (создать ER-диаграмму)
- [ ] Определить стратегию двойной работы старой/новой моделей в переходный период
- [ ] Решить: словари (дисциплины, классы дронов, каналы) — seed в миграции или редактируемые админом?
- [ ] Согласовать формы официальных протоколов (PDF-экспорт, печать)
- [ ] Зафиксировать соответствие ролей RBAC ↔ должностям ГСК

### Фаза 1 — Справочники и базовая модель (2-3 недели)

**Цель:** загрузить статические данные правил в БД.

1. Миграция: создать таблицы `disciplines`, `race_systems`, `race_formats`, `age_groups`, `video_channels`, `drone_specs`
2. Seed: 8 дисциплин, требования к каждому классу из Таблицы 10, возрастные группы
3. Расширить `users.judge_role`, `users.qualification_category`
4. Расширить `pilots` — пол, дата рождения, возрастная группа, мед.допуск
5. API: GET-эндпойнты для справочников
6. UI: страница «Справочники» (только просмотр)
7. Тесты: все справочники загружены корректно, дисциплина имеет валидные технические требования

### Фаза 2 — Команды и заявки (2-3 недели)

**Цель:** регистрация команд и подача заявок по образцам Приложений 1-3.

1. Таблицы: `teams`, `team_members`
2. Расширить `competitions`: `discipline_id`, `playoff_size`, `age_group_id`, `entry_fee`
3. Эндпойнты: создание команды, добавление/удаление участников
4. Логика капитана: один на команду, определяет состав вылета (2 чел.)
5. **Предварительная заявка** — webhook + UI-форма (Приложение 2)
6. **Окончательная заявка** — UI-форма (Приложение 3) с подписями: представитель, рук. рег. федерации, рук. ОИВ, врач
7. **Согласие родителей** для несовершеннолетних — загрузка PDF/скана (Приложение 1)
8. Бизнес-правила: для возр. группы 10-17 проверка возраста на день начала; для остальных — на календарный год
9. Тесты: команда из 3 чел. (1 пилот + 1 механик + 1 запасной), валидация заявок, отказ без мед.допуска

### Фаза 3 — Комиссия по допуску (1-2 недели)

**Цель:** реализовать процедуру допуска участников и техосмотр дронов.

1. Таблицы: `admissions`, `documents`, `equipment_inspections`, `drones`
2. Workflow: заявка → загрузка документов → проверка ГСК → решение (допущен / отказ + причина)
3. Технический контроль дрона: судья проверяет соответствие `drone_specs`, помечает passed/rejected
4. Валидаторы: вес ≤ X, размер диагонали в диапазоне, частота 5.8 ГГц, наличие файл-сейфа (флаг), наличие светодиодов
5. UI: «Кабинет судьи комиссии» — список заявок с действиями
6. Эндпойнт: PDF-протокол комиссии (печать решений)
7. Тесты: отказ при отсутствии страховки, отказ при превышении веса дрона, успешный допуск с полным комплектом

### Фаза 3.5 — Хранилище загружаемых документов (технический блок)

**Цель:** безопасное локальное хранение PDF/сканов для заявок, допусков и согласий.

1. Backend dependency: `multer` для multipart upload
2. Конфигурация upload-папки:
   - `DOCUMENTS_ROOT` — явный override для DEV/QA
   - Windows installer / bundled exe — `%APPDATA%/FeoSport2/uploads/`
   - DEV fallback — `data/uploads/`
3. Installer: `setup-db.ps1` создаёт upload-папку и пишет `DOCUMENTS_ROOT` в `.env`
4. API: загрузка, список, метаданные, download, delete для `documents`
5. Безопасность: whitelist MIME, лимит размера, SHA-256, path traversal guard
6. Тесты: unit для резолва upload path + integration для Documents API

### Фаза 4 — Этапы и распределение групп (3-4 недели)

**Цель:** автоматизация турнирной сетки по табл. 3, 4, 6.

1. Таблицы: `stages`, `groups`
2. Квалификация — два варианта (выбор организатора в Положении):
   - Прохождение N кругов за минимальное время
   - Максимум кругов за установленное время
3. Автогенератор групп для **системы 2/4**:
   - 1/8 финала (32 пилота → 8 групп по 4)
   - 1/4 финала (16 → 4 группы по 4)
   - Полуфинал (8 → 2 группы по 4)
   - Финал (4)
4. Автогенератор для **системы 4/8** (для симулятора):
   - 1/4 финала (32 → 4 группы по 8)
   - Полуфинал (16 → 2 группы по 8)
   - Финал (8)
5. Жеребьёвка квалификации (рандомизация состава групп)
6. Логика «по результатам N-го этапа в следующий проходят M пилотов»
7. Обработка неявки: подтягивание следующего из квалификационного списка
8. UI: визуализация турнирной сетки (bracket view) — уже частично есть, переделать под группы
9. Тесты: корректность распределения по табл. 3 и 6, отказ при попытке сгенерировать финал без полуфинала

### Фаза 5 — Проведение вылетов и хронометраж (3-4 недели)

**Цель:** замена текущих heats на полноценные вылеты с круговым таймингом.

1. Таблицы: `laps`, `falsestarts`, `reflights`
2. WebSocket-события:
   - `flight_start` (готовность пилотов подтверждена)
   - `lap_complete` (по сигналу хронометража)
   - `falsestart` (с указанием виновника)
   - `flight_end` (DNF / финиш / DSQ)
   - `reflight_requested` (по правилу 5.5.7.3.x)
3. Правила перелёта (автомат + ручной):
   - Фальстарт → перелёт всей группы
   - Столкновение в зоне старта (до 1-го элемента) → перелёт всей группы
   - Столкновение после 1-го элемента по вине пилота → перелёт без виновника
   - Проблема с видеосигналом (правило 5.5.7.3.8) — заявка от пилота, 5-мин окно
4. Управление **видеоканалами**: назначение канала пилоту, обнаружение конфликта, swap-логика
5. Интеграция с системой хронометража (автоматическая или ручная — открытый интерфейс ввода)
6. UI: «Пульт хронометриста» — табло активного вылета с кнопками круга/финиша/нарушения
7. UI: «Экран зрителя» — live leaderboard с временами по кругам
8. Тесты: подсчёт лучшего круга, корректность перелётов, broadcast результатов всем подключённым судьям

### Фаза 6 — Командные особенности (2-3 недели)

**Цель:** пит-стоп, обмен пилот/механик, замена дрона.

1. Таблицы: `pit_stops`
2. События: `pit_stop_entered`, `pit_stop_exited`, `battery_replaced`, `drone_replaced`, `pilot_mechanic_swap`
3. Бизнес-правило: на 50% дистанции — обязательный обмен пилот↔механик (предупреждение если не выполнен)
4. Учёт аккумуляторов (количество, статус заряда) — судья пит-стоп зоны принимает АКБ в хранение
5. UI: «Кабинет судьи пит-стоп зоны» — учёт замен и таймеров
6. Тесты: корректность фиксации пит-стопов, валидация swap-обязательства

### Фаза 7 — Симулятор (2-3 недели)

**Цель:** поддержка цифровых дисциплин.

1. Таблицы: `disconnects`
2. Поле в `competitions`: `simulator_software_name`, `simulator_version`, `simulator_server_settings_json`
3. Логика дисконнектов:
   - Все пилоты — переигровка по решению ГСК
   - Один пилот многократно — техническое поражение
   - Настраиваемые пороги (Положение): таймаут ожидания, кол-во попыток
4. Формат online: добавление визуальной верификации участников (фото/видео)
5. Интеграция со сторонним симулятором — открытый webhook для приёма результатов (если симулятор поддерживает)
6. UI: настройки симулятора в карточке соревнования
7. Тесты: дисконнект-сценарии, переигровка, техническое поражение

### Фаза 8 — Штрафы и протесты (1-2 недели)

**Цель:** официальная лестница санкций.

1. Таблицы: `penalties`, `protests`
2. Типы штрафов (5.10): устное замечание, предупреждение, лишение баллов, техническое поражение, дисквалификация
3. Workflow протеста:
   - Подача в течение 5 минут после гонки
   - Поле со ссылкой на пункт правил
   - Решение главного судьи (письменно, на инфо-стенд)
   - Окончательность решения
4. UI: «Подача протеста» (для представителя), «Рассмотрение» (для главного судьи)
5. Уведомления: WebSocket-событие при поступлении/решении протеста
6. Тесты: протест после 5 мин отклоняется, дисквалификация аннулирует результат

### Фаза 9 — Подсчёт баллов и победители (1-2 недели)

**Цель:** корректное распределение мест по правилам 5.17.

1. Логика баллов финального этапа:
   - 4-чел. группа: 3/2/1/0 баллов
   - 8-чел. группа: 4/3/2/1/0
   - +1 балл за 2 победы (для 4/4-системы)
   - Дуэль при равенстве баллов (отдельный вылет)
2. Логика командных мест:
   - 1-е по времени, 2-4 по кругам/времени (после первой команды + 5 мин)
3. Места с 5-го и ниже — настраиваемо в Положении (config)
4. Расчёт командного зачёта
5. Тесты: дуэль при ничьей, +1 балл за 2 победы, корректность мест 5+

### Фаза 10 — Протоколы и отчётность (2-3 недели)

3. Подписи: главный судья + главный секретарь (электронная — через хэш + JWT-аудит)

## 5. Стратегия миграции

**Не переделываем за один релиз.** Подход:

1. Версия `v2.0-alpha`: справочники + расширенные пилоты + команды (Фазы 1-2)
   — работает параллельно со старым функционалом
2. Версия `v2.0-beta`: комиссия допуска + этапы + хронометраж (Фазы 3-5)
   — пилотный запуск на реальном соревновании 1 дисциплины
3. Версия `v2.0`: командные + симулятор + протесты + протоколы (Фазы 6-10)
4. Версия `v2.1`: полная переделка UI + расширенные роли (Фазы 11-12)
5. Версия `v2.2`: безопасность и аудит (Фаза 13)

## 7. Открытые вопросы (требуют решения до старта)

1. **Хранение документов** — локально (для оффлайн-инсталляции) или облако? Сейчас инсталлятор предполагает локальный PostgreSQL.
2. **Интеграция с системой хронометража** — какие железные системы используются на реальных соревнованиях? Открытый API?
3. **Электронная подпись протоколов** — достаточно JWT-аудита или нужна КЭП (квалифицированная электронная подпись)?
4. **PDF-вёрстка протоколов** — точное соответствие образцам или функциональный эквивалент?
5. **Многосоревновательность** — система обслуживает одно соревнование за раз или несколько параллельно? Сейчас архитектура поддерживает несколько `competitions`, но UI заточен под одно.
6. **Интернационализация** — только русский (по приказу Минспорта) или мультиязычность для международных гостей?
7. **Зрительский режим** — отдельный публичный URL без аутентификации?
8. **Видеотрансляция и анализ цифрового следа** (5.8.4) — интеграция с внешними сервисами или собственная подсистема?
